# 🎬 AI 漫剧/短剧全自动工业级生产系统 —— 核心架构与深度剖析报告

本报告旨在深度解析并梳理【AI 漫剧自动化生产系统】的演进历程，详细剖析 **01-初始版本** 与 **02-二代版本** 的核心流程与设计精髓，并全面展示 **03-融合版本** 的当前架构。
最关键的是，本报告重点盘点了**当前融合版本中遗漏的、亟待从前两个版本中吸收的底层设计细节与核心机制（TODO清单）**，作为后续代码与架构全面修复、重构的终极指导手册。

---

## 🎯 一、 我们的目标与愿景

**一句话概括核心目标：**
将一部几十万甚至数百万字的长篇网络小说（TXT格式），通过完全自动化的管线，一章一章地转化为**高质量、可播放的漫剧/短剧视频**。

为了实现这一目标，系统必须解决 AI 视频生成领域最致命的问题：
*   **长线连载不崩盘**：角色不能换脸、衣服不能穿帮、场景不能瞬间漂移。
*   **剧情不断档**：宏观上把握百章剧情走势，微观上卡准每一集的动作与情绪转折。
*   **半自动工业化流水线**：摒弃“一键盲盒”的不可控性，在关键节点（如资产定妆、分镜剧本）引入**人机协同（Human-in-the-loop）**与本地锁死机制。

## 🔍 二、 01-初始版本：长线叙事的“资产死锁”方案（V1深度拆解）

**01版本的核心价值在于：它是针对长篇小说“连载一致性”开出的特效药。** 它没有把重点放在最终的视频渲染，而是放在了前期的“资产盘点”与“上下文文脉”的滚动控制上。

### 1. 核心工作流与滚动机制

01版本创造性地设计了**“双轨滚动机制”**与**“三级漏斗管线”**，把庞大的小说切碎并稳稳接住：
*   **第一层（宏观）：百章级全量滚动环（100章切片）**
    *   **动作**：提取百章“宏观世界线航海图”。
    *   **细节机制**：严格控制大模型输出字数必须在输入原文字数的 **10% ~ 18%** 之间，低于10%视为偷懒熔断，高于18%视为冗余。强制分为 5 个 20 章的大卷，包含核心大事件编年史和战力/版图演进账本。
*   **第二层（中观）：十章级精加工（10章切片）**
    *   **动作**：生成本轮 10 章的“全局剧情脉络与视听控制总纲”。
    *   **细节机制**：利用上一期的总纲作为“质量血统”进行代际传递。
*   **第三层（微观）：单章解剖与排产**
    *   **动作**：以 1 章为单位，进行分集、提取纯中文分镜、榨取纯音频对白。
    *   **细节机制**：场记接力棒——本集的第一镜必须继承上一集的绝杀尾镜，绝对禁止场景漂移穿帮。

### 2. 十二大资产盘点系统（“人口普查”）

在百章级管线中，强制 DeepSeek 只做“名词拉单子”，把万物归入 **12 个极其严格的 XML 标签** 中，不带任何长篇废话。这是后续所有一致性的地基：
1. `<角色清单>`：建演员表，锁脸型。
2. `<服装与造型清单>`：锁衣服，防一件衣服穿到大结局。
3. `<场景清单>`：确定置景。
4. `<时间与时代清单>`：控昼夜、调色。
5. `<核心事件清单>`：卡集数高潮点。
6. `<道具与线索清单>`：锁定武器、法宝、关键物证。
7. `<技能与特殊能力>`：规范特效（如斗气颜色）。
8. `<系统UI与特效字幕>`：独立UI贴图。
9. `<势力与组织清单>`：规范群演服装与图腾。
10. `<非人生物与坐骑>`：怪兽控制。
11. `<世界设定清单>`：设定战力逻辑，防模型胡编。
12. `<环境音与卡点清单>`：提供给 FFmpeg 卡点音效用。

### 3. 本地定妆与 WebUI 盲盒试镜
01版本引入了“选角大厅”。大模型根据外貌描述跑出多模态三视图盲盒，人类在 WebUI 点击确认后，系统在本地物理硬盘生成 `角色_王林_少年初始态_全方位定妆大账.md` 与 `.png`。这就是不可篡改的“生死簿”，下游生图必须无脑调用这张脸。

## 🛠️ 三、 02-二代版本：工程化的“流水线执行引擎”（V2深度拆解）

如果说01版本是“编剧和场记”，那 **02版本就是真正的“制片厂和后期机房”**。02版本的核心价值在于其**强大的底层工程架构、多API适配能力以及自动合成链路**。

### 1. 五层叙事建模（解决AI精分问题）
02版本纠正了最致命的“叙事层级错位”问题（之前让大模型同时干编剧、分镜和视觉），确立了影视工业标准的五层结构：
*   **L0 原著文本**
*   **L1 分集切分**
*   **L2 叙事蓝图（Episode MD - 关键补充层）**：负责冲突规划、情绪曲线、节奏控制。
*   **L3 剧本层**：负责具体的对白与动作。
*   **L4 分镜层**：纯粹将剧本翻译成视觉画面拆解与提示词。

### 2. 统一 API 网关与适配层
*   **大一统网关**：所有大模型（GPT/DeepSeek）、图像（Kling/Flux/Runway）、TTS配音、音乐引擎的调用，都必须穿过这个网关。
*   **并发与重试**：内建指数退避重试、异步轮询（针对视频生成这种慢任务）以及信号量并发控制。

### 3. 任务调度与状态机大脑
定义了整条生产流水线的状态流转大脑：从 `PENDING` -> `LLM_GENERATING` -> `WAITING_REVIEW`（人工审核锁）-> `GENERATING_IMAGE` -> `GENERATING_VIDEO` -> `COMPOSING` -> `DONE`。支持断点续传与失败回滚。

### 4. 自动化剪辑合成层
02版本将生图、视频、配音、配乐整合。特别是设计了全局可配置的**画幅比例控制（9:16 / 16:9 / 1:1）**，并包含了基于此的动态字幕位置推算和 Ken Burns（推拉摇移）镜头的自动化应用。

## 🧬 四、 03-融合版本：当前的架构全景图

03-融合版本尝试将 01 和 02 的精髓结合起来，提出了目前的 **“五层两横切”总体架构**：

1.  **第一层：宏观战略层**（题材识别 + 01版的十二大资产盘点 + 百章航海图）。
2.  **第二层：中观编排层**（十章总纲 + 资产参数细化 + 02版的L2叙事蓝图与L3剧本生成）。
3.  **第三层：微观生产层**（单章分镜拆解 + 01版声音车间/视觉车间分类 + 提示词二次精炼防崩坏）。
4.  **第四层：素材执行层**（02版的统一API网关 + 多策略一致性控制）。
5.  **第五层：合成输出层**（Ken Burns静态转动态 + FFmpeg音画拼接）。
*   **横切一：控制中枢**（02版的状态机大脑）。
*   **横切二：数据底座**（Markdown文件持久化与WebUI）。

虽然框架搭建起来了，但这只是一个**“骨架”**。在实际代码和文档中，大量决定成败的“血肉细节”并未成功从 01 和 02 迁移过来。

## 🚨 五、 核心指导：融合版本亟待补充与吸收的细节清单（TODO红皮书）

这是本报告的**重中之重**！后续的修复与重构，必须将以下 01 和 02 版本的顶级机制 100% 还原并实装到目前的 03 融合版代码中。

### 🔴 1. 01版资产锁定与物理状态落盘机制的缺失
*   **[遗漏细节] 物理定妆库死锁**：融合版本目前提到了定妆，但缺少 01 版本中强制的 **本地 `.md` 文件与 `.png` 参考图绑定落盘**机制（例如 `03-全局核心角色定妆库/角色_王林_少年初始态_全方位定妆大账.md`）。必须在前端 WebUI 实现状态监控：“未定妆”、“跑盲盒转圈”、“天选之脸锁定”，并靠本地文件作为唯一的真相来源（Single Source of Truth）。
*   **[遗漏细节] 严苛的 XML 正则剥离法**：01版本第一层只允许大模型输出带有 `<角色清单>` 等标签的“纯名词”，并用 Python 正则强行提取。融合版本似乎对资产提取的格式限制过于宽松，容易导致大模型在此处生成废话，破坏下游脚本的稳定性。必须补回正则表达式严格清洗机制。

### 🔴 2. 01版“滚雪球”字数流控与防崩机制的缺失
*   **[遗漏细节] 10% - 18% 字数对齐公式**：01版明确要求百章航海图的输出必须落在输入生肉文本字数的 10%~18% 区间，否则直接熔断。融合版本丢掉了这一防偷懒/防注水机制。必须在 Prompt 和验证脚本中重新加回动态字数区间熔断器。
*   **[遗漏细节] 场记接力棒（尾镜接首镜）**：01版微观单章处理时，强制“本集第一镜必须继承上一集的绝杀尾镜”。这个机制在融合版本的微观生产层未见详细的系统级约束，必须在代码层面注入上一集的最后画面提示词作为当前集的上下文。

### 🔴 3. 02版叙事层与分镜层解耦的不彻底
*   **[遗漏细节] L2叙事蓝图与L3剧本的刚性拆分**：虽然融合版提到了“叙事蓝图与剧本”，但在实际流水线中，仍有重合或跳跃的情况。必须严格贯彻 02版的理念：大纲只管集数和钩子，叙事蓝图（L2）专门写情绪曲线和冲突，剧本（L3）专门写纯对白和行为，分镜（L4）纯粹做机位和视觉描述。绝不允许在提示词阶段再去补充剧情逻辑。

### 🔴 4. 02版工程执行层的高级配置缺失
*   **[遗漏细节] 画幅比例的全局枚举驱动**：02版本后期的补丁明确要求将画幅（`9:16`, `16:9`, `1:1`）作为全局 `config` 的 `ASPECT_RATIO_MAP` 字典下发，它不仅影响分辨率，还自动决定字幕位置（例如0.80 或 0.88）和 Ken Burns 的主轴（纵向或横向）。目前的融合版本没有把这个动态派生配置贯彻到全链路。
*   **[遗漏细节] API 指数退避与信号量**：融合版的素材执行层只画了架构图，但缺少 02版中强调的 `asyncio.Semaphore` 控制并发量以及防止请求过载的退避重试算法。必须在网关层补全这块工程代码。

### 🔴 5. 纯音频“脱水榨干”逻辑的强化
*   **[遗漏细节] 声音车间的绝对剥离**：01 版规定了大模型做“声肉脱水”时，要以公式“1秒=4个字”进行测算，并无脑映射到本地 `Voice ID`。融合版在音频生成的描述上过于宽泛。需在 TTS 前置处理中补回过滤视觉环境废话、纯提纯对白及测算秒数的硬编码逻辑。
