# 📂 第一部分技术账本

## ——滑窗输入数据拓扑、跨章吞噬边界与创世块特判

本部分核心解决：**"大模型开机前，本地 Python 脚本怎么把原材料洗干净并打包"**。

### 1. 输入的【参考内容】（大模型看什么）

* **素材 A：【后顾接力棒】**
* *正常状态*：读取 `第X-1章_流水线生产总执行派工单.json` 尾部字段。
* *第 1 章冷启动特判*：不读磁盘，直接在内存中强行注入`创世块全中文伪派工单变量`（含假台词钩子"大幕拉开，万物初始"及假视觉钩子"电影级全黑转场"，防止系统由于找不到第 0 章而崩溃）。

* **素材 B：【中观导演手册】**：从 `XX-XX全局剧情脉络与视听控制总纲.md` 中截取出的当前章节微观里程碑与情绪锚点文本。
* **素材 C：【前瞻预判灯塔】**：`第X+1章.txt` **100% 毫无保留、零删减、零截断的全量原著正文本**（扔给大模型让它自己带着智能去过滤噪点，寻找跨章反转爆点）。
* **素材 D：【系统刚性教条】**：常驻在脚本底层的静态 System Prompt，包含"1秒视频=4汉字配音"、"单集脱水字数锁死在 240~360 字"的数学公式和全中文标签合同规则。

### 2. 输入的【处理内容】（对什么东西开刀）

* **【第 X 章.txt 原始正文】**：当前正在排产的、未经过任何加工的原始网络小说生肉正文（体量一般在 3000 ~ 8000 字不等）。

### 3. 产出的【内容】

* **【第 X 章_切剩的纯净正文】**：一段纯文本字符串。这是 Python 脚本根据上一章的结束句标记，在当前章开头物理切掉被吃剩的 600 字（或若干字）之后，**剩下的、真正属于本期排产的纯净正文**。

### 4. 产出内容有【什么作用】

* **彻底斩断"剧情死循环复读机"**：确保上一章已经拍过、吞噬过的剧情，在这一章的开头被百分之百物理抹除，绝对不会重复发给大模型，下游生图和 TTS 绝不翻车。
* **合流组装**：这个切剩的纯净正文，将与上述 4 大参考内容一起，在内存中合流组装成一个高内聚的"终极分集请求包（Payload Context）"，一键砸给大模型 API。

### 5. 以【什么方式、形式、路径保存】

* **保存形式**：不落盘，作为内存临时字符串变量（Memory Buffer）直接常驻在 Python 的运行内存中。
* **物理路径**：无物理路径。组装成 API 请求体（Request Body）后直接向大模型接口发送，发送完毕随进程销毁，零硬盘读写损耗。

# 📂 第二部分技术账本

## ——全中文标签协议输出合同与本地 Python JSON 自动化编译引擎

本部分核心解决：**"大模型把干货吐出来后，本地 Python 脚本怎么物理切片并组装高精 JSON"**。

### 1. 输入的【参考内容】

* **【第一部分合流发送的终极请求包】**（即第一部分汇总的系统规则 + 中观大纲 + 上章JSON + 本章切剩正文 + 下一章全本）。

### 2. 输入的【处理内容】

* **大模型（DeepSeek）根据输入请求，在内存中实时演算、通读、并流式返回的【全中文标签纯文本数据流】**。

### 3. 产出的【内容】

本部分是整个流水线物料落地的终点，一次性物理产出两类、共计 $N+1$ 个文件：

1. **核心总指挥蓝图**：`第X章_流水线生产总执行派工单.json`
2. **单集物理原著切片**：$N$ 个独立的 `ep_XX_raw.txt` 纯原著正文文件（$N$ 为大模型动态计算出来的总集数）。

### 4. 产出内容的【按什么要求产出】（刚性合规红线）

* **标签无噪红线**：大模型吐出的文本必须严格被 `<开始分集图谱>`、`<开始分集_N>`、`文本终点锚点：` 等全中文标签锁死。**绝对不准夹带大括号、中括号、Markdown 井号等任何格式污染。**
* **物理闭环红线**：相邻两集的 `文本终点锚点` 在原著字符串里必须是连续的，中间不能丢掉一个标点符号。
* **跨章吞噬标志红线**：如果大模型最后一集抢了下一章的肉，必须在标签内强制声明 `是否跨章吞噬：是`，并把下一章被吃掉的终点句子写在锚点上。

### 5. 产出内容有【什么作用】

* **作为 Python 物理剪刀的图纸**：Python 根据大模型在纯文本标签里指定的中文结束句子，在原著里定位，咔咔几刀把大 TXT 物理切成几百字一集的独立文本块，**完成了"网络小说长篇生肉"向"短剧单集标准工业物料"的降维转变**。
* **锁死长线因果钢印**：JSON 中记录的 `音频台词钩子` 与 `画面视觉钩子`，将作为钢印死死扣住下一章的起点，**彻底根治章节切换时"主角长相突变、衣服场景瞬间平移"的行业绝症**。
* **下游高频减负**：下游负责提炼提示词和对白的大模型，每次开机只需要读切好的、几百字的 `ep_XX_raw.txt`，注意力集中度提升一万倍。

### 6. 以【什么方式、形式、路径保存】

* **保存形式**：**本地标准化结构化多级文件夹形式**。
* **目录树路径定盘规范**：
在项目的排产大目录下，Python 脚本会现场为当前章节创建专属的微观排产夹，长相如下：

```text
02-大纲/01-仙逆/142-仙逆1411-1420章节大纲/第1415章微观排产区/
├── 第1415章_流水线生产总执行派工单.json      # 100%全中文映射、本地Python编译组装的终极高精JSON
├── ep_01/
│   └── ep_01_raw.txt                       # 第一集纯净原著小说生肉片段（TXT形式）
├── ep_02/
│   └── ep_02_raw.txt                       # 第二集纯净原著小说生肉片段（TXT形式）
└── ep_03/
    └── ep_03_raw.txt                       # 第三集（触发跨章吞噬后，Python在后台自动强行缝合出来的无缝高潮原著TXT）

```
