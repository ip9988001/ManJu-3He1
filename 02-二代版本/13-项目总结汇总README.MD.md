# 🎬 AI 漫剧/短剧自动化生产系统 —— 全流程详解

## 一句话概括

**把一篇小说扔进去，系统自动帮你变成一部可以播放的短剧视频。** 中间经过题材识别 → 剧本拆分 → 分镜脚本 → AI画图 → AI做视频 → AI配音 → AI配乐 → 自动剪辑合成 → 成片输出。

---

## 🔄 完整生产流水线（8大阶段）

```
小说原文
  ↓
① 题材识别 + 视觉风格加载
  ↓
② 分集大纲生成
  ↓
③ 叙事蓝图（Episode MD）+ 剧本生成  ← 补充层
  ↓
④ 逐镜分镜脚本生成
  ↓
⑤ 提示词精炼（图像 + 视频）
  ↓
⑥ 多媒体素材生成（图片/视频/配音/音乐/音效）
  ↓
⑦ 自动剪辑合成
  ↓
⑧ 成片输出
```

**关键：每一步之间都有人工审核节点，不是"一键盲盒"，而是"半自动工业化"。**

---

## 📌 阶段一：题材识别 + 视觉风格加载

### 这一步在做什么？

系统读取你扔进来的小说原文，**自动判断这是什么题材**（玄幻修仙？现代都市霸总？悬疑惊悚？古风言情？等等），然后**加载该题材对应的视觉风格词库**。

### 具体产出什么？

| 产出物 | 说明 |
|--------|------|
| **题材标签** | 比如"玄幻修仙"，决定整个剧的视觉基调 |
| **视觉词库** | 该题材的核心视觉元素，如"飞剑流光、云雾仙山、法宝灵光" |
| **色彩基调** | 比如"冷青、雪白、暗金为主色调" |
| **光影风格** | 比如"云层体积光、冷月逆光" |
| **镜头语言** | 比如"低机位仰拍、中近景压迫构图" |
| **服装道具词库** | 该题材常见的服饰和道具 |
| **环境场景词库** | 该题材常见的场景 |
| **禁用混入元素** | 比如玄幻题材里禁止出现"现代高楼、汽车、枪械" |

### 功能和作用？

**这一步相当于给整个项目"定调"。** 就像拍电影前先确定"我们拍的是武侠片还是科幻片"，不同题材的视觉风格完全不同。有了这个"风格锚点"，后续所有生成环节都围绕它来，避免画面风格混乱。

### 人工审核点

✅ 建议审核 —— 确认题材判断是否准确、视觉词库是否合适

---

## 📌 阶段二：分集大纲生成

### 这一步在做什么？

把小说原文**拆分成适合短视频平台观看节奏的分集剧本大纲**。不是照搬小说章节，而是按照"短剧节奏"重新编排。

### 具体产出什么？

| 产出物 | 说明 |
|--------|------|
| **集数 + 集名** | 比如"第一集：雪夜残钟" |
| **核心冲突** | 这一集的核心矛盾是什么 |
| **关键场景** | 3-5个关键场景描述 |
| **情绪走向** | 开场→中段→高潮→结尾的情绪曲线 |
| **出场角色** | 本集出场的角色及其状态 |
| **角色资产初稿** | 首次出场角色的完整外貌描述（年龄、五官、发型、体型、服装、标志物） |
| **本集视觉锚点** | 本集最核心的视觉元素 |
| **结尾钩子** | 让观众想继续看下一集的悬念/反转 |
| **分镜拆解建议** | 建议拆几个镜头、怎么拆 |

### 功能和作用？

**这一步相当于"编剧部"的工作。** 把长篇小说变成一集一集的短剧，每集都有冲突、高潮、悬念。短剧的核心是"节奏"——前3秒抓住观众，中段增加信息，高潮情绪爆发，结尾留钩子。

### 短剧节奏规则

- **前3秒**：直接展示危机、死亡、背叛、强冲突
- **开场**：明确困境（主角受伤、失去重要人物、被羞辱）
- **中段**：增加信息（线索、反派痕迹、误会、能力异常）
- **高潮**：情绪爆发（复仇、反杀、崩溃、觉醒）
- **结尾**：续看钩子（悬念、反转、身份揭露）

### 人工审核点

✅ 必须审核 —— 确认分集节奏、角色设定、冲突设计是否合理

---

## 📌 阶段三：叙事蓝图（Episode MD）+ 剧本生成

### ⚠️ 这是补充设计层，解决"层级错位"问题

原来的系统直接从"大纲"跳到"分镜"，中间缺少两个关键层。补充后变成：

```
大纲 → 叙事蓝图（Episode MD）→ 剧本（Script）→ 分镜
```

### 3.1 叙事蓝图（Episode MD）—— 相当于"制片人/总策划"

| 产出物 | 说明 |
|--------|------|
| **本集目标** | 本集核心冲突是什么 |
| **情绪曲线** | 起→承→转→合 的情绪走向图 |
| **场景列表** | Scene 1 引入 → Scene 2 冲突 → Scene 3 升级 → Scene 4 反转 → Scene 5 结尾钩子 |
| **角色状态锁定** | 主角当前心理状态、反派动机状态、配角功能状态 |
| **信息密度分配** | 哪些关键信息必须在本集出现 |
| **对白风格约束** | 本集语言风格（古风文雅？现代口语？冷冽简洁？） |

### 3.2 剧本层（Script）—— 相当于"编剧"

| 产出物 | 说明 |
|--------|------|
| **逐场景剧本** | 每个场景的完整对白、动作描写、情绪节奏 |
| **场景调度** | 谁在什么位置、做什么动作 |
| **情绪节奏标注** | 每句台词的情绪强度 |

### 功能和作用？

**这两层解决了"AI同时当编剧+分镜师+视觉设计师导致质量不稳定"的问题。**

- 叙事蓝图 = 确保故事结构扎实、节奏正确
- 剧本 = 确保台词精彩、角色不漂移、情绪连续
- 分镜只需要做"视觉拆解"这一件事，输入更精准，产出更稳定

### 人工审核点

✅ 必须审核 —— 叙事蓝图和剧本都需要人工确认

---

## 📌 阶段四：逐镜分镜脚本生成

### 这一步在做什么？

把剧本**逐镜头拆解成可以直接喂给AI图像/视频/配音模型的结构化数据**。每个镜头（分镜）都包含画面、动态、台词、音乐、音效的完整信息。

### 每个分镜包含什么？（共26个字段）

| 字段 | 作用 | 通俗解释 |
|------|------|----------|
| **场景编号** | 排序标识 | 比如"001""002" |
| **分镜标题** | 这个镜头叫什么 | 比如"雪夜醒来" |
| **画面叙事目的** | 这个镜头为什么要存在 | 比如"建立宗门覆灭的压抑氛围" |
| **角色名称** | 画面里有谁 | 比如"萧玄" |
| **角色外貌** | 角色长什么样 | 完整描述年龄、五官、发型、体型、服装 |
| **角色锚定词** | 角色的"身份证" | 一串固定的关键词，确保AI每次画同一个人 |
| **角色情绪** | 角色现在什么心情 | 比如"虚弱、迷茫、压抑的痛楚" |
| **角色动作** | 角色在做什么 | 比如"单膝跪在雪地中，一手撑着断剑" |
| **服装道具** | 穿什么、拿什么 | 比如"玄色破损长袍，符文断剑" |
| **场景环境** | 在哪里、什么天气 | 比如"黄昏雪夜，破败山门，坍塌石阶" |
| **光影构图** | 光从哪来、怎么构图 | 比如"冷蓝雪光为主，远处残火橙色逆光" |
| **镜头语言** | 用什么镜头拍 | 比如"低机位中近景，浅景深" |
| **画面风格** | 什么画风 | 比如"国风玄幻漫剧，电影级厚涂质感" |
| **文生图提示词** | 🎨 直接喂给AI画图的完整提示词 | 80-180字的画面描述 |
| **反向提示词** | 告诉AI不要画什么 | 比如"角色变脸，多余手指，肢体扭曲" |
| **视频动态提示词** | 🎬 直接喂给AI做视频的提示词 | 描述镜头怎么动、画面有什么微动态 |
| **运动限制** | 视频里禁止什么动作 | 比如"禁止角色起身、奔跑、换脸" |
| **镜头时长** | 这个镜头几秒 | 比如"五秒" |
| **转场建议** | 怎么切换到下一个镜头 | 比如"从黑场淡入" |
| **台词配音** | 🎤 角色说什么台词 | 比如"萧玄：是谁……灭了我青玄宗。" |
| **情绪标签** | 配音用什么情绪 | 比如"虚弱" |
| **角色音色** | 用什么声音配 | 比如"低沉冷冽的青年男声" |
| **BGM提示词** | 🎵 背景音乐什么风格 | 比如"低速国风史诗配乐，古琴低音" |
| **音效提示词** | 🔊 有什么环境音效 | 比如"寒风声，细雪落地声，远处残钟低鸣" |
| **一致性备注** | 后续要注意什么 | 比如"后续萧玄必须保持半束黑发、脸侧浅血痕" |
| **人工审核提示** | 哪里需要人确认 | 比如"确认断剑是否作为全剧固定视觉符号" |

### 功能和作用？

**这一步是整个系统的"翻译器"——把人类的故事语言翻译成AI能理解的生产指令。** 每个分镜就是一张"生产工单"，包含了生产图片、视频、配音、音乐、音效所需的全部信息。

### 人工审核点

✅ 必须审核 —— 每个分镜的字段都需要人工确认

## 📌 阶段五：提示词精炼（图像 + 视频）

### 这一步在做什么？

对阶段四产出的"文生图提示词"和"视频动态提示词"进行**二次精炼优化**，让它们更适合AI模型理解。

### 5.1 文生图提示词精炼

**优化方向：**
- 补全角色细节（年龄、五官、发型、体型、服装、标志物）
- 补全环境细节（时间、地点、天气、背景物体）
- 补全光影信息（光源方向、色彩基调、明暗关系）
- 补全镜头信息（景别、机位、角度、景深）
- 删除空泛词（"很美""很帅""很震撼"）
- 确保角色一致性（不改变角色核心外貌特征）

**质量标准：** 提示词必须是"客观画面描述"，不是"对模型的请求"。长度控制在120-260个中文字符。

### 5.2 视频动态提示词精炼

**核心安全规则（防止AI视频崩坏）：**

| 禁止动作 | 为什么禁止 | 替代方案 |
|----------|-----------|----------|
| 快速奔跑 | 容易导致人物结构漂移 | 镜头加速推进，衣物更明显飘动 |
| 飞速冲向 | 容易导致画面变形和瞬移 | 推镜头，中景转近景 |
| 猛地跳起 | 容易导致肢体崩坏 | 身体重心轻微变化，衣袂上扬 |
| 高速旋转 | 容易导致脸部和背景崩坏 | 低速环绕镜头 |
| 连续打斗 | 多动作难稳定 | 武器光效闪烁、指节收紧、眼神变冷 |
| 变身换装 | 破坏角色一致性 | 光效增强但服装和五官不变 |

**运动强度校准：**
- 悲伤/平静 → 低强度（微表情、呼吸、雪花飘落）
- 紧张/悬疑 → 中强度（缓慢推镜、阴影移动、灯光闪烁）
- 愤怒/战斗 → 高强度（推镜头稍快、衣物强烈飘动、能量光效增强）

### 功能和作用？

**这一步是"质量保险"。** 直接用原始提示词生成图片/视频容易出问题（角色变脸、画面崩坏、风格不一致），精炼后的提示词更稳定、更专业。

### 人工审核点

✅ 必须审核 —— 精炼后的提示词需要人工确认

## 📌 阶段六：多媒体素材生成

### 这一步在做什么？

**把分镜里的所有"提示词"变成真正的素材文件。** 这是系统中调用外部API最多的阶段。

### 6.1 图像生成（文生图）

- **输入：** 文生图提示词 + 反向提示词 + 角色锚定词
- **输出：** 一张图片文件
- **调用的API：** Kling（可灵）、Flux、Seedream、Stable Diffusion、DALL-E等
- **关键参数：** 画幅比例（9:16竖屏/16:9横屏/1:1方形）、种子值（seed）、参考图

### 6.2 视频生成（图生视频）

- **输入：** 生成的图片 + 视频动态提示词 + 运动限制 + 镜头时长
- **输出：** 一段3-6秒的视频
- **调用的API：** Kling、Runway、Seedance、Luma、PixVerse等
- **关键：** 异步任务，需要轮询等待结果

### 6.3 配音生成（TTS）

- **输入：** 台词文本 + 情绪标签 + 角色音色描述
- **输出：** 音频文件（mp3/wav）
- **调用的API：** ElevenLabs、MiniMax、Fish Audio、Edge-TTS、阿里云TTS等
- **关键功能：**
  - **角色音色绑定：** 每个角色绑定固定音色，全剧一致
  - **情绪驱动：** 根据情绪标签调整语速、语调、稳定性
  - **降级机制：** ElevenLabs挂了自动切MiniMax，再挂切Edge-TTS

### 6.4 背景音乐生成（BGM）

- **输入：** BGM提示词（音乐风格、节奏、情绪、乐器）
- **输出：** 音乐文件
- **调用的API：** Suno、Udio、MusicGen等
- **关键：** 时长需要与视频匹配

### 6.5 音效生成（SFX）

- **输入：** 音效提示词（环境音、动作音、转场音）
- **输出：** 音效文件
- **实现方式：** AI生成 或 从音效库检索匹配
- **关键功能：** 时间轴控制（音效在什么时间点播放）

### 功能和作用？

**这一步是"素材工厂"。** 所有AI模型在这里工作，把文字描述变成看得见、听得到的素材。

### 自动降级机制

系统内置了fallback链路，某个API挂了自动切换到备选：
```
图像：Kling → Flux → Seedream
视频：Kling → Runway → Seedance → Luma
配音：ElevenLabs → MiniMax → Edge-TTS
```

## 📌 阶段七：自动剪辑合成

### 这一步在做什么？

**把零散的素材文件组装成完整的视频片段。** 这是"最后一公里"。

### 7.1 素材下载与缓存

- 把云端生成的图片、视频、音频下载到本地缓存

### 7.2 静态图 → 动态视频（Ken Burns效果）

- 如果只生成了图片还没做视频，用Ken Burns效果让静态图片"动起来"
- 效果包括：缓慢推进、横向平移、纵向平移、缓慢缩放
- **画幅自适应：** 竖屏以纵向运动为主，横屏以横向运动为主

### 7.3 音画同步

- 把配音音频和视频画面对齐
- 确保"角色说话时嘴在动"

### 7.4 BGM混音

- 把配音、BGM、音效三层音频混合
- BGM音量通常压到20%，配音为主

### 7.5 字幕生成

- 根据台词生成ASS格式字幕
- 字幕位置根据画幅自动调整

### 7.6 单分镜合成

- 把一个分镜的所有素材（视频+配音+BGM+音效+字幕）合成一个完整片段

### 7.7 整集拼接

- 把一集里所有分镜片段按顺序拼接
- 处理转场效果（淡入淡出、切黑等）

### 功能和作用？

**这一步是"后期制作部"。** 把AI生成的零散素材变成真正可以播放的短剧视频。

## 🧠 横向支撑系统（贯穿全流程）

### 1. API 网关层

**作用：** 所有外部API调用的统一入口

- **统一接口：** 不管调Kling还是Runway，调用方式一样
- **自动重试 + 指数退避：** 失败了自动重试，每次等待时间翻倍
- **并发控制：** 用信号量控制同时调用几个API
- **成本记录：** 记录每次API调用花了多少钱
- **可插拔适配器：** 新增一个API服务商只需要写一个适配器文件

### 2. 任务调度 + 状态机

**作用：** 控制整个生产流水线的"大脑"

**状态流转：**
```
待处理 → LLM生成中 → 等待人工审核 → 已确认
→ 图像生成中 → 视频生成中 → 音频生成中 → 合成中 → 生产完成
```

**核心功能：**
- **断点续传：** 系统崩了可以从上次中断的地方继续
- **人工审核Hook：** 到了审核节点自动暂停等人确认
- **角色一致性策略分发：** 自动选择用Seed、LoRA还是FaceSwap来保持角色一致
- **并发调度：** 多个分镜可以同时生产

### 3. 角色一致性控制

**作用：** 确保同一个人物在不同镜头里长得一样

**三种策略：**

| 策略 | 原理 | 适用场景 |
|------|------|----------|
| **Seed** | 固定随机种子，让AI生成相似图片 | 简单场景、短剧 |
| **LoRA** | 训练角色专属模型 | 长剧、需要高一致性 |
| **FaceSwap** | 先生成再换脸 | 高动态场景、动作戏 |

### 4. WebUI 控制台

**作用：** 人机交互的操作界面（用Gradio构建）

**6个功能Tab：**

| Tab | 功能 |
|-----|------|
| **项目管理** | 创建/打开/删除项目，配置模型、画幅、题材 |
| **角色资产库** | 管理角色外貌、音色、参考图、LoRA |
| **剧本工作台** | 小说输入 → Prompt生成 → 分镜编辑 → MD导出 |
| **素材生产台** | 选择分镜 → 一键生产图片/视频/配音/BGM |
| **合成输出台** | 预览 → 合成 → 导出成片 |
| **成本监控** | 查看LLM/图像/视频/语音各项成本 |

### 5. Markdown 文件系统（SSOT）

**作用：** 整个系统的"唯一真理来源"

- LLM输出 → 解析 → 写入Markdown → 人工审核确认 → 后续所有环节从Markdown读取数据
- 每个分镜一个`.md`文件，包含全部26个字段
- 所有素材文件路径也记录在Markdown中

## 🔧 技术栈总结

| 层级 | 技术 | 作用 |
|------|------|------|
| 后端API | FastAPI | REST接口、任务调度入口 |
| 前端UI | Gradio Blocks | 人工审核、编辑、预览 |
| 配置 | pydantic-settings + .env | API Key、模型、并发配置 |
| HTTP调用 | httpx.AsyncClient | 异步调用所有外部API |
| 状态存储 | SQLite（可升级PostgreSQL） | 任务状态、断点续传、成本记录 |
| 文件中间层 | Markdown | 唯一真理文件 |
| 调度 | asyncio + 状态机 | 并发、重试、限流、断点恢复 |
| 合成 | FFmpeg / MoviePy | 字幕、音画同步、视频合成 |
| LLM | DeepSeek / Kimi / GPT-4o | 生成剧本、分镜、提示词 |

---

这套系统的核心设计理念就是：**不追求"一键出片"的魔法，而是构建一条可控、可审核、可重试的工业化生产线。** AI负责生成，人类负责审核和决策，两者配合才能稳定批量生产高质量漫剧。
