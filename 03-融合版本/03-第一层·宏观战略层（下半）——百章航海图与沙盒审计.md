# 融合方案·第03部分：第一层·宏观战略层（下半）——百章航海图与沙盒审计

---

## 一、百章级全量滚动处理

### 1.1 功能说明

【取自系统2】每 100 章为一个大周期，让大模型生成一份宏观世界线航海图——相当于整部剧的"总导演手册"。它锁定了这 100 章内的所有重大事件、战力变化、伏笔暗线和视听风格，确保长线连载不崩盘。

### 1.2 输入模式

| 轮次 | 输入内容 | 说明 |
|------|----------|------|
| 首轮（001-100章） | 100 章原文纯文本 + 投入总字数 Win | 纯净小说生肉，作为基准 |
| 后续轮（101-200章...） | 上一轮航海图 + 新 100 章原文 | 上一轮航海图作为"因果律锁链"防止脱节 |

### 1.3 动态字数流控铁律

**【取自系统2】** 防止大模型偷工减料或废话连篇。

**数学公式：**

$$W_{in} \times 10\% \le W_{out} \le W_{in} \times 18\%$$

- $W_{in}$：本轮投入的 100 章原始正文总字数
- $W_{out}$：生成的航海图总字数

**实例核算（以 40 万字输入为例）：**
- 下限：40,000 × 10% = 4,000 字（低于此判定偷工减料，熔断重跑）
- 上限：40,000 × 18% = 7,200 字（高于此强制截断，防止信息冗余）

### 1.4 产出物

**文件名：** `XX-章节XXX-XXX宏观世界线航海图.md`

### 1.5 四大电影级维度

#### 维度一：百章级核心大事件编年史（占 60%）

| 字段 | 说明 |
|------|------|
| 宏观卷切分 | 严格以 20 章为一个宏观剧情卷（Arc），100 章切分为 5 个大卷 |
| 卷级信息锁 | 每卷的名称、起止章节、核心矛盾、高潮定格点 |
| 章级结构化梗概 | 逐章 250-350 字硬核事实，拒绝主观评价 |

**章级梗概只准写：**
- 这一章主角做出了什么关键决策、杀了谁、拿到了什么
- 这一章场景发生了什么物理迁移
- 这一章引入了哪个核心配角，该配角对主角的态度

**章级梗概禁止写：**
- "这一章很精彩""作者写得不错"等主观评价
- 重复前文已有的信息

#### 维度二：金手指升级与战力天花板锚定线（占 15%）

| 字段 | 说明 |
|------|------|
| 主角战力滚动轨迹 | 精确到具体章节，卡死每次量变与质变的瞬间（如第7章凝气一层，第22章凝气三层） |
| 金手指状态图层 | 金手指在 100 章内的所有形态演进、能量异变、生图中必须呈现的物理流光视觉差 |

#### 维度三：全局悬念与暗线引线图（占 15%）

| 字段 | 说明 |
|------|------|
| 新埋引线 | 抓出本轮 100 章中所有"初次提及、但至第 100 章结束仍未爆发"的长期潜伏危机、仇恨线与隐藏势力，标明初次出现的精准章节 |
| 历史核销账本 | 从第二轮开始触发：大模型必须对照上一轮航海图，明确列出上一轮留下的哪几条暗线在本轮被正式回收、引爆或产生推进 |

#### 维度四：阶段性视听调色盘与情绪代差（占 10%）

| 字段 | 说明 |
|------|------|
| 视觉全局 LUT 规范 | 配合 5 个 20 章大卷的剧情迁移，给出全局调色方案：主色调 Hex 倾向、画面光质、环境物理微动指令 |
| 配乐与音效主调 | 为不同剧情阶段锁定全局 BGM 的主导伴奏乐器与频响质感 |

### 1.6 尾数熔断规则

针对小说总章数不是 100 整数倍的情况：

| 规则 | 条件 | 处理方式 |
|------|------|----------|
| 尾数 ≤ 30 章 | 如总共 1725 章 | 最后一轮直接打包 125 章统一处理，字数基准自动调整 |
| 尾数 > 30 章 | 如总共 1787 章 | 最后 87 章单独启动最后一轮，照常调用上一轮航海图作为记忆底座 |

### 1.7 红线约束

| 红线 | 说明 |
|------|------|
| 字数流控 | 低于 10% 熔断重跑，高于 18% 强制截断 |
| 章级只写事实 | 禁止主观评价，只准输出物理行为 |
| 暗线必须追踪 | 从第二轮开始，必须核销上一轮的暗线 |
| 调色必须精确 | 必须给出 Hex 色值，不准用"暖色调""冷色调"等模糊词 |

### 1.8 人工审核点

✅ 建议审核——确认大事件编年史是否准确、战力变化是否合理、伏笔暗线是否完整、调色方案是否合适。

### 1.9 在上下游中的作用

- **上游依赖**：12 大资产盘点结果 + 题材识别结果
- **下游服务**：为十章视听控制总纲提供"宏观灯塔"（因果链、伏笔、调色依据）

## 二、百章沙盒审计系统

### 2.1 功能说明

【取自系统2】每 100 章结束后，对沙盒中积累的所有资产卡片进行去重、归宗、升级判定。解决"大模型自己会造同义词"的问题。

### 2.2 百章增量临时资产表

**文件名：** `XXXX-XXXX百章增量临时资产表.md`

**生命周期：**

| 阶段 | 动作 | 说明 |
|------|------|------|
| 诞生 | Python 检测到跨入新百章大关时自动创建 | 初始化为空表 |
| 积累 | 每轮新产生的资产名字追加写入 | 只进不出 |
| 终结 | 百章审计完毕后物理删除或归档 | 释放全局检索空间 |

**内部格式（脱水版实名制账本）：**

```markdown
# 1401-1500百章增量临时资产表

## [SANDBOX_LAYER: 01-核心角色]
* 实名标识: [王林_元婴初期态]
  - 核心视觉: 皮肤转为晶莹、神色冷酷、白发飘动。
* 实名标识: [李慕婉]
  - 核心视觉: 面色苍白、身穿淡雅素雪绢衣、神态温柔。

## [SANDBOX_LAYER: 02-服装造型]
* 实名标识: [战神殿内门血袍]
  - 核心视觉: 暗红色粗麻、左胸绣有金色战字盾徽。

## [SANDBOX_LAYER: 05-道具线索]
* 实名标识: [天逆珠_四纹形态]
  - 核心视觉: 珠体表面四道飞鸟划痕微亮、散发木属性青光。
```

### 2.3 背景白名单注入机制

**【取自系统2】** 每当十章滚动环启动时，Python 脚本在大模型"开机"前执行白名单注入。

**工作流：**
1. Python 提取当前章节段（如 1411-1420），定位到所属的百章增量表
2. 全量读取该表文字，转化为纯文本字符串 `[Sandbox_Whitelist_Context]`
3. 将该变量无缝焊接至发送给大模型的 System Prompt 底部

**注入 Prompt 的核心指令：**

```markdown
【最高死命令】
在你提炼12大资产时，你必须首先核对下方提供的《百章增量临时资产表》。

【源头净化去重算法规则】
1. 绝对实名复用原则：如果本轮资产与表中已有资产语义匹配度超过80%，
   必须无条件沿用表里的标准名字！
2. 严禁二次发明：复用的资产只列名字，严禁重新编写细胞级参数卡片。
3. 形态升级与新增判定：只有发生重大变异才允许输出新版本卡片。
```

### 2.4 审计 Agent 两步过滤漏斗

**【取自系统2】** 当 100 章结束时，Python 唤醒本地审计 Agent。

```
【百章暂存区：~300个原始资产卡片】
       │
       ▼
 ┌───────────┐
 │  第一道闸 │ ── 本地名称与特征字符串硬匹配 ──→ (0 Token，斩杀80%)
 └───────────┘
       │
       ▼ 【剩余争议资产】
 ┌───────────┐
 │  第二道闸 │ ── 低成本LLM 1对1盲审 ──→ (精准判定，极低Token)
 └───────────┘
```

#### 第一道闸：本地名称与特征字符串硬匹配（0 Token 消耗）

| 检测类型 | 逻辑 | 结果 |
|----------|------|------|
| 完美撞车 | 文件名与历史库完全一致 | `EXACT_REUSE`，直接拦截 |
| 基础名撞车 | 去掉版本号后匹配（如 飞剑_v2 vs 飞剑_v1） | `SUSPECTED_EVOLUTION`，送第二步 |
| 模糊相似度 | Levenshtein 相似度 > 0.7 | `SUSPECTED_SYNONYM`，送第二步 |
| 全新资产 | 以上都没命中 | `PURE_NEW`，直接通过 |

#### 第二道闸：低成本 LLM 1对1 盲审

**使用模型：** DeepSeek-V4-Flash（非思考模式，极低成本）

**盲审 System Prompt 核心规则：**

```markdown
# 剧组资产铁面审计官指令

判定决策树：
1. COMPLETE_REUSE（完全复用）：核心视觉高度一致，只是多了过场描写
2. ASSET_EVOLUTION（形态升级）：发生重大不可逆版本跃迁
   - 角色：年龄暴长、突破导致容貌异变、断臂战损
   - 道具：法宝升级、流光由低阶变高阶、彻底破碎
3. SYNONYM_ERASE（近义抹除）：名字不同但本质是同一个物理积木

输出格式：必须严格JSON
{"action": "COMPLETE_REUSE|ASSET_EVOLUTION|SYNONYM_ERASE",
 "reason": "一句话理由",
 "recommended_name": "标准名称"}
```

### 2.5 三种判定的物理执行

| 判定 | 物理动作 |
|------|----------|
| `COMPLETE_REUSE` | 删除当前暂存卡片，路由指针指向历史已有文件 |
| `ASSET_EVOLUTION` | 保留卡片，重命名为标准格式（如 `char_wanglin_v2_18岁态.md`），打上晋升标识 |
| `SYNONYM_ERASE` | 删除当前暂存卡片，在内存路由表登记强制重定向（如"恒岳派灰袍"→"恒岳派灰色弟子服"） |

### 2.6 资产晋升机制

所有通过审计的 `PURE_NEW` 和 `ASSET_EVOLUTION` 资产：
1. Python 执行 `shutil.move`，从各章节的 `临时资产沙盒暂存区/` 拔出
2. 整齐码放到全局唯一的 `历史权威资产库/` 对应大类目录下
3. 正式拥有历史编制，供后续千万章随时调用

### 2.7 JIT 虚拟路标集编译引擎

**【取自系统2】** Agent 回溯这 100 章内的每一个十章级文件夹，为它们各自编译一个专属的 `00-本轮资产绝对路径路标集.json`。

**JSON 格式：**

```json
{
  "chunk_scope": "1411-1420",
  "compiled_timestamp": "2026-06-14T21:00:00Z",
  "active_asset_routing_table": {
    "01-核心角色": {
      "王林": "历史权威资产库/01-核心角色/char_wanglin_v3_元婴态.md",
      "李慕婉": "历史权威资产库/01-核心角色/char_limuwan_v1.md"
    },
    "05-道具线索": {
      "天逆珠": "历史权威资产库/05-道具线索/prop_tianbizhu_v4_four_lines.md"
    }
  }
}
```

**下游调用性能：** O(1) 时间——下游生图脚本开机只读 1 个 JSON 文件，内存一键装载字典，直接查表找到任何资产的绝对路径。

### 2.8 红线约束

| 红线 | 说明 |
|------|------|
| 审计必须触发 | 100 章结束必须启动审计，不得跳过 |
| 近义词必须归宗 | 不允许同一资产有两个名字并存 |
| 升级必须有版本号 | 形态升级的资产必须带 v1/v2/v3 版本号 |
| 路标集必须编译 | 每个十章文件夹必须有 JSON 路标集 |

### 2.9 人工审核点

✅ 可选审核——确认近义词判定和升级判定是否正确。对于关键角色的形态升级，建议人工确认。

### 2.10 在上下游中的作用

- **上游依赖**：各轮沙盒暂存区中的原始资产卡片
- **下游服务**：为单章生产提供 O(1) 资产检索；为后续百章提供历史记忆基础
