
# 🎬 AI 漫剧自动化生产系统（第二套）—— 全流程详解

## 一句话概括

**把一部长篇网络小说（几十万字甚至几百万字）扔进去，系统自动帮你一章一章地变成可以播放的漫剧短视频。** 和第一套系统不同，这套系统的核心设计理念是**"长线连载不崩盘"**——角色不换脸、衣服不穿帮、场景不漂移、剧情不断档。

---

## 🔄 完整生产流水线（5大阶段）

```
长篇小说原文（几百章 TXT 文件）
  ↓
① 第一阶段：百章级宏观资产盘点（"人口普查"）
  ↓
② 第二阶段：十章级中观精加工 + 百章沙盒审计（"制片人会议"）
  ↓
③ 第三阶段：微观单章解剖与分集排产（"编剧+导演"）
  ↓
④ 第四阶段：全局角色/场景定妆与本地锁死（"选角+定妆照"）
  ↓
⑤ 第五阶段：声音车间 + 视觉车间 + 提示词编译（"拍摄+后期"）
```

**核心特色：双轨滚动前进 + 全中文 Markdown 文件系统 + 本地定妆照锁脸**

## 📌 阶段一：百章级宏观资产盘点（"人口普查"）

### 这一步在做什么？

系统读取小说的前 10 章（约 5 万字），让 DeepSeek 进行**"扫盲式人口普查"**——只拉名词清单，不产生任何长篇描述。

### 具体产出什么？（12 大资产科目表）

| 序号 | 资产类型 | 通俗解释 | 在漫剧里的作用 |
|------|----------|----------|----------------|
| 1 | **核心角色** | 所有有台词、有动作的人物名字 | 建立"演员表"，锁定角色脸型 |
| 2 | **服装与造型** | 特殊服饰、门派制服、面具 | 防止一件衣服穿到大结局 |
| 3 | **场景空间** | 所有物理地点（室内/室外/虚拟空间） | 确定"置景"，生成背景 |
| 4 | **时间与时代** | 时代背景、时间跨度、昼夜状态 | 决定调色风格和白天黑夜控制 |
| 5 | **核心事件** | 推动剧情转折的大事件 | 规划分集剧本，卡准每集高潮 |
| 6 | **道具与线索** | 武器、法宝、关键物证 | 保证关键道具不走样 |
| 7 | **技能与特殊能力** | 功法招式、超能力 | 规范特效视觉效果 |
| 8 | **系统UI与特效字幕** | 属性面板、弹窗提示语 | 制作独立视觉贴图 |
| 9 | **势力与组织** | 门派、家族、财阀 | 批量生成群演服装、规范图腾 |
| 10 | **非人生物与坐骑** | 妖兽、神兽、宠物 | 专门的怪兽类生图控制 |
| 11 | **世界设定** | 战力等级、货币、地理界域 | 规范升级链，防止模型胡编 |
| 12 | **环境音与卡点** | 标志性声响、动作卡点 | 指导音效时间戳对齐 |

### 功能和作用？

**这一步相当于给整部剧建立"户口本"。** 长篇连载最危险的就是"世界观脱节"（前面说修仙后面变科幻）、"角色服装前后不一"（第一章穿黑袍第五十章变白袍）、"设定吃书"（前面说练气期后面忘了）。通过第一步的名词清单把这些隐患全部锁死。

### 人工审核点

✅ 必须审核 —— WebUI 上展示 12 个标签池，人工删掉不重要的龙套、添加漏掉的名词

### 产出物

- 12 个独立的 `.md` 文件（每个对应一个资产类型）
- 存入 `global_assets.json`

## 📌 百章级全量滚动处理（"宏观世界线航海图"）

### 这一步在做什么？

每 100 章为一个大周期，让大模型生成一份**宏观世界线航海图**——相当于整部剧的"总导演手册"。

### 具体产出什么？（四大电影级维度）

| 维度 | 占比 | 内容 |
|------|------|------|
| **百章级核心大事件编年史** | 60% | 以 20 章为一卷，逐章抠出关键行为动作（每章 250-350 字） |
| **金手指升级与战力天花板锚定线** | 15% | 主角战力每次变化的精确章节、金手指形态演进 |
| **全局悬念与暗线引线图** | 15% | 新埋的伏笔、被回收的暗线 |
| **阶段性视听调色盘与情绪代差** | 10% | 每卷的主色调 Hex、光影风格、BGM 乐器 |

### 功能和作用？

**这一步是"长线不崩盘"的核心保障。** 没有这个航海图，大模型在处理第 500 章时根本不知道第 50 章埋了什么伏笔。有了它，整部剧的因果链、战力体系、视觉风格都有了"总纲"。

### 字数控制铁律

输出字数必须在输入字数的 **10%-18%** 之间。比如输入 40 万字，输出必须在 4-7.2 万字之间。

## 📌 阶段二：十章级中观精加工 + 百章沙盒审计（"制片人会议"）

### 这一步在做什么？

每 10 章为一轮，做两件事：
1. **生成"十章视听控制总纲"** —— 相当于这一段戏的"分场导演手册"
2. **生成 12 大类资产的详细参数卡片** —— 把阶段一的"名词清单"升级为"细胞级参数"

### 2.1 十章视听控制总纲包含什么？

| 维度 | 内容 | 通俗解释 |
|------|------|----------|
| **分章剧情里程碑** | 逐章的核心推进任务 + 分集切分 | 这一章要拍几集、每集的悬念钩子是什么 |
| **线索与战力流转图** | 逐章监控角色战力变化、道具流转 | 谁在第几章变强了、法宝到了谁手里 |
| **全局情绪曲线** | 10 章内的情绪震荡数值（-5 到 +5） | 哪几集是低谷、哪几集是高潮 |
| **最后一幕视听定格** | 最后一镜的精确物理描述 | 下一轮开机时从哪个画面接 |

### 2.2 12 大资产的"细胞级参数"是什么意思？

不是简单写"王林是个少年"，而是拆到**解剖学级别**：

**以核心角色为例，6 大参数模块：**

| 模块 | 包含什么 | 举例（韩立） |
|------|----------|-------------|
| **基础身份** | 名字、生命周期阶段、性别、阵营 | 韩立，少年记名弟子期，男，七玄门 |
| **面部解剖学** | 肤色、眼部、发型、面部锚点 | 黝黑哑光皮肤、单眼皮狭长眼、灰色麻绳扎马尾 |
| **躯体形态** | 身高、体型、站姿剪影 | 1.62米精瘦、站立时微驼、双手随时戒备 |
| **动态神情** | 微表情、高频小动作、爆发态 | 嘴角下抿、思考时摩挲小瓶、杀意时面部死寂 |
| **声线参数** | 生理年龄感、声带纹理、语速 | 变声期少年音、音质干枯偏冷、每分钟180字 |
| **特写抓手** | 该拍他哪里 | 布满老茧、指甲缝有草药黑汁的右手特写 |

**以服装为例：** 精确到面料的十字编织纹理、完全吸光的哑光质感、交领右衽的图层嵌套拓扑、战损痕迹。

**以场景为例：** 三层景物布局（前景枯藤/中景青石台/背景纯黑阴影）、唯一主光源角度、丁达尔效应、空间混响模型。

### 2.3 百章沙盒审计（每 100 章触发一次）

当 100 章的 10 轮滚动处理完成后，启动**本地审计 Agent**，用两步过滤漏斗清理资产：

**第一步：本地名称硬匹配（0 Token 消耗）**
- 完全撞车 → 判定为"复用"，直接拦截
- 基础名撞车（去掉版本号） → 判定为"疑似升级"，送第二步
- 字符串相似度 > 70% → 判定为"疑似近义词"，送第二步
- 以上都没命中 → 判定为"全新资产"

**第二步：低成本 LLM 盲审**
- 针对"疑似升级"和"疑似近义词"，用低成本模型做 1 对 1 对比
- 三种判定：完全复用（删掉）、形态升级（保留并改名）、近义抹除（删掉并重定向到历史名称）

### 功能和作用？

**这一步解决"大模型自己会造同义词"的问题。** 比如第 10 章叫"恒岳派灰色弟子服"，第 50 章大模型可能自己发明一个"恒岳派灰袍"。审计 Agent 会自动检测并归宗到同一个标准名称。

### 最终产出

- `00-本轮资产绝对路径路标集.json` —— 一个 JSON 文件，包含本轮所有资产的精确文件路径
- 下游生图脚本读这一个 JSON 就能在 O(1) 时间内找到任何资产

## 📌 阶段三：微观单章解剖与分集排产（"编剧+导演"）

### 这一步在做什么？

**逐章处理**——把一章小说（3000-8000字）切成 1-4 集短剧（每集 60-80 秒），并为每集生成完整的生产工单。

### 3.1 输入组装（5 大素材合流）

| 素材 | 来源 | 作用 |
|------|------|------|
| **系统刚性教条** | Python 底层常数 | 锁死"1秒=4汉字"、单集 240-360 字 |
| **中观导演手册** | 十章视听控制总纲 | 告诉大模型这章是大战章还是过渡章 |
| **后顾接力棒** | 上一章的 JSON 派工单 | 确保从上一章最后一镜无缝接续 |
| **当期原料** | 当前章 TXT（已切掉上章吃剩的部分） | 正文素材 |
| **前瞻预判灯塔** | 下一章 TXT（100%全量） | 让大模型知道未来走向，设计最佳悬念点 |

### 3.2 跨章吞噬机制

当一章内容不够凑满一集时，大模型可以"跨章吞噬"——抢吃下一章的开头部分。系统通过"吞噬终点句子"来精确标记吃到了哪里。

### 3.3 创世块（Genesis Block）

处理第一章时，因为没有"上一章"，系统在内存中硬编码一组假数据：
- 上一集台词钩子："大幕拉开，万物初始。"
- 上一集画面钩子："电影级全黑转场"
- 防止系统因找不到第 0 章而崩溃

### 3.4 大模型输出格式（全中文标签协议）

**严禁输出 JSON 或 Markdown**，只用中文尖括号标签：

```
<开始分集图谱>
总集数：3

<开始分集_1>
文本终点锚点：墨大夫突然转过身来，眼中寒芒暴涨。
音频台词钩子：墨大夫，这是你逼我的！
画面视觉钩子：局部特写，主角右手狠狠拍在储物袋上，绿芒喷薄而出。
当前临界资产：王林_少年初始态, 墨大夫
<结束分集_1>
...
<结束分集图谱>
```

### 3.5 Python 物理切肉

Python 脚本根据大模型指定的"文本终点锚点"句子，在原著 TXT 中精确定位，物理切割成独立的 `ep_01_raw.txt`、`ep_02_raw.txt`...

### 功能和作用？

**这一步是"长篇小说→短剧单集"的关键转换。** 把几千字的小说章节变成几百字一集的标准化工业物料，每集都有精确的台词钩子（结尾悬念）和画面钩子（定格画面）。

### 产出物

```
第X章微观排产区/
├── 第X章_流水线生产总执行派工单.json    ← 核心总指挥
├── ep_01/
│   └── ep_01_raw.txt                    ← 第一集原著切片
├── ep_02/
│   └── ep_02_raw.txt
└── ep_03/
    └── ep_03_raw.txt
```

## 📌 阶段四：全局角色/场景定妆与本地锁死（"选角+定妆照"）

### 这一步在做什么？

在整部剧正式开机生图前，**为核心角色和标志性场景生成"定妆照"**，并存入本地硬盘。

### 具体流程

1. **WebUI 选角大厅** —— 输入角色外貌描写，网页自动向云端 API 轰出多张"多角度三视图"
2. **人工盲盒筛选** —— 鼠标点击选中最符合原著气质的那张
3. **本地物理存盘** —— 自动下载为 `.png` 文件，存入 `03-全局核心角色定妆库/`
4. **生成定妆大账 `.md`** —— 记录角色的黄金 Prompt 母本、种子号、本地图片路径

### 定妆照怎么用？

下游生图时，Python 脚本**像发微信附件一样**，把本地硬盘里的定妆照原图连同英文提示词一起打包发给线上 API。线上模型"看着这张脸来画"，角色相似度直接飙到 90% 以上。

### 功能和作用？

**这一步终结了线上模型的"脸盲症"。** 没有定妆照，每次生图出来的脸都不同；有了定妆照，全剧几千个镜头的王林都是同一张脸。

### WebUI 联动

账本 `.md` 文件里有一个状态字段：`未定妆` / `盲盒渲染中` / `已定锁`。WebUI 根据这个字段决定显示灰色剪影、转圈还是高清大图。

## 📌 阶段五：声音车间 + 视觉车间 + 提示词编译（"拍摄+后期"）

### 这一步是整个系统的"生产车间"，包含三个并行工作的"车间"：

---

### 5.1 声音车间（步骤 03）

**在做什么？** 把原著分集文本"声肉脱水"——剥离所有视觉废话，只保留 100% 纯粹的配音内容。

**输入：**
- 分集原著生肉 `ep_01_raw.txt`
- 上一步的控制常数（台词钩子、临界资产）
- 十章中观视听控制总纲
- 全中文标签合同（System Prompt）

**输出：** `01集_全角色配音与情绪调教派工单.md`

**产出内容举例：**
```
<开始配音行_1>
发音角色：旁白
情绪音调：【低沉引入】冷调悬疑，语速放慢15%
有效声音内容：荒山，古庙。暴雨如注，砸在青苔乱石之上。
<结束配音行_1>

<开始配音行_2>
发音角色：王林_少年初始态
情绪音调：【极度惊恐】呼吸声急促，带有一丝少年特有的颤音
有效声音内容：谁？！是谁在庙后说话？！
<结束配音行_2>
```

**核心红线：**
- 有效声音内容里**严禁夹带动作描述**（不允许"王林：（惊恐地退后两步）谁？！"）
- 角色名字必须用资产表里的标准实名
- 零对白场景强制全量判定为"旁白"
- 最后一行配音必须与台词钩子 100% 逐字对齐

**功能和作用？**
1. **自动计算全集总时长**：总字数 ÷ 4字/秒 = 视频秒数
2. **驱动 TTS 批量配音**：Python 根据角色名字匹配 Voice ID，根据情绪参数调整语速语调

### 5.2 视觉车间 · 房间一：分镜导演办公室（步骤 04 前半）

**在做什么？** 纯粹从**电影导演的角度**，把这一集切成一镜一镜的画面设计。不考虑 AI 生图的英文提示词，只用中文电影行话。

**输入：**
- 场记接力棒（上一集最后一镜的画面描述）
- 中观总纲（当前 10 章的合法场景和色调）
- 音频派工单（配音行号 = 时间轴依据）
- 当期原著生肉

**输出：** `01集_电影级镜头画面构图设计.md`

**产出内容举例：**
```
### [分镜镜头_01]
* 对应声音：[配音时间轴节点_01]
* 画面景别：大远景（Extreme Long Shot）
* 画面视觉构图描述：暴雨如注的深夜，恒岳派后山的破败古庙在黑暗中风摇雨晃。
  一道惨白闪电劈开云层，瞬间将古庙残破瓦砾照得一片惨白。
* 出场核心资产标签：后山破庙, 特大暴雨, 闪电狂风

### [分镜镜头_02]
* 对应声音：[配音时间轴节点_02]
* 画面景别：特写（Close-up）
* 画面视觉构图描述：15岁少年王林背靠开裂朱红木柱，满脸混着冷汗的雨水，
  双瞳因极度惊恐急剧收缩，死死盯着破庙门前的一团阴影。
* 出场核心资产标签：王林_少年初始态, 破庙内部开裂木柱
```

**核心红线：**
- 第一镜必须继承上一集的尾镜画面（场记接力棒）
- 绝对不去重（同一角色连续出现 3 次 = 3 个独立镜头）
- 资产名称必须用标准实名

**功能和作用？** 这是**人眼对账的核心**——你在生图前可以像看剧本一样检查每一镜的画面设计。

### 5.3 视觉车间 · 房间二：技术画师提示词编译（步骤 04 后半）

**在做什么？** 把房间一的纯中文分镜**翻译编译成英文提示词**，并指定本地定妆照的物理路径。

**输入：**
- 房间一产出的中文分镜
- 全局资产路径账本（角色定妆照的本地硬盘路径）
- 动态全局画风表

**输出：** `01集_Flux画质特写高精提示词.md`

**产出内容举例：**
```
### [生图控制节点_01]
* 附加本地参考图: None
* 线上生图核心Prompt: (Cinematic long shot, masterpiece, 8k), 
  dark teal and gray tone, an ancient temple in heavy thunderstorm --ar 16:9
```

## 🧠 横向支撑系统（贯穿全流程）

### 1. 双轨滚动工作流

| 轨道 | 触发时机 | 处理跨度 | 核心目的 |
|------|----------|----------|----------|
| **轨 A：宏观资产盘点** | 每 10 章触发 | 10 个 TXT 文件 | 调用 DeepSeek 批量拉名词清单，更新资产库 |
| **轨 B：微观单章精制** | 轨 A 锁定后自动激活 | 1 个 TXT 文件 | 逐章出分镜、生图、配音、合成视频 |

**运转逻辑：**
```
轨 A：锁定 01-10 章资产 → 轨 B：逐章精制第1章→第2章→...→第10章
→ 轨 A：增量更新 11-20 章资产 → 轨 B：逐章精制第11章→...→第20章
→ 无限套娃
```

### 2. 全中文 Markdown 文件系统

**为什么用 `.md` 不用 JSON？**
- Python 解析极简：看到 `* 发音角色：` 就 `.split("：")[1]` 拿走角色名
- 人工改账无感：你在记事本里改了字，Python 下次读到的就是改后的版本
- 可视化：在文件管理器里一眼看清整部剧的所有资产

**Obsidian 联动：** 通过 `[[王林_少年初始态]]` 双链标签，在 Obsidian 图谱里看到这个角色在哪些分镜里出现过。

## 🆚 两套系统的核心差异

| 维度 | 第一套系统 | 第二套系统 |
|------|-----------|-----------|
| **目标** | 通用短剧生产线 | 长篇连载漫剧生产线 |
| **输入** | 一篇小说 | 几百章 TXT 文件（几十万~几百万字） |
| **角色一致性** | Seed / LoRA / FaceSwap | 本地定妆照 + 多模态真图传参 |
| **数据格式** | XML 标签 + Markdown | 全中文尖括号标签 + Markdown + JSON |
| **资产管理** | 一次性生成 | 百章沙盒审计 + 增量滚动 + 历史权威资产库 |
| **长线防崩** | 状态机断点续传 | 宏观航海图 + 中观总纲 + 跨章吞噬 + 场记接力棒 |
| **审核方式** | WebUI 审核 | WebUI + Obsidian 双链可视化 |

---

这套系统的核心理念就是：**不追求一步到位的魔法，而是用"粗→细→精"三层递进的工业流程，把一部长达几百万字的小说，一章一章、一镜一镜地变成高质量漫剧。** 每一层都有人工审核节点，每一步都有防崩机制，确保长线连载几千集下来角色不换脸、衣服不穿帮、剧情不断档。
