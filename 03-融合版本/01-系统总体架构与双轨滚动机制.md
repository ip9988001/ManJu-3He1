# 融合方案·第01部分：系统总体架构与双轨滚动机制

---

## 一、系统一句话定义

把一部长篇网络小说（几十万字到几百万字、几百章 TXT 文件）扔进去，系统自动帮你一章一章地变成可以播放的漫剧短视频。角色不换脸、衣服不穿帮、场景不漂移、剧情不断档。

### 2.2 各层职责定义

| 层级 | 名称 | 核心职责 | 类比 |
|------|------|----------|------|
| 第一层 | 宏观战略层 | 锁定全剧世界观、角色、场景、风格的"户口本" | 制片人 + 美术总监 |
| 第二层 | 中观编排层 | 锁定每 10 章的剧情节奏、情绪曲线、资产参数 | 编剧总监 + 选角导演 |
| 第三层 | 微观生产层 | 逐章切集、逐镜设计、逐句配音 | 现场导演 + 场记 |
| 第四层 | 素材执行层 | 调用 AI 模型生成图片/视频/音频 | 摄影师 + 录音师 |
| 第五层 | 合成输出层 | 把零散素材拼装成可播放的视频 | 后期剪辑师 |
| 横切一 | 控制中枢 | 管理全流程状态、审核、成本、异常 | 监制 + 质检 |
| 横切二 | 数据底座 | 所有数据的存储、检索、传递 | 剧组档案室 |

### 2.3 各层之间的数据流关系

```
第一层产出 → 百章航海图 + 12大资产库
    ↓ 作为"宏观灯塔"注入第二层
第二层产出 → 十章总纲 + 细胞级资产参数 + 叙事蓝图 + 剧本
    ↓ 作为"中观导演手册"注入第三层
第三层产出 → 分集派工单 + 配音派工单 + 分镜设计 + 英文提示词
    ↓ 作为"生产工单"注入第四层
第四层产出 → 图片文件 + 视频文件 + 音频文件
    ↓ 作为"素材物料"注入第五层
第五层产出 → 成品 MP4 视频

横切一 → 贯穿所有层，管理状态流转和人工审核
横切二 → 贯穿所有层，提供数据存储和检索
```

## 三、双轨滚动前进机制

### 3.1 为什么需要双轨？

长篇小说有几百章，如果一次性全部处理：
- 大模型上下文窗口装不下
- 一次生成太多内容质量会下降
- 前面的角色设定到后面会被遗忘

所以必须**分批处理**，但又不能"每批独立"，否则前后会脱节。

**解决方案：双轨滚动——宏观盘点和微观精制交替前进。**

### 3.2 双轨定义

| 轨道 | 名称 | 触发时机 | 处理跨度 | 核心目的 |
|------|------|----------|----------|----------|
| 轨A | 宏观资产盘点 | 项目初始化，或当前10章视频全部导出完毕 | 10个TXT文件 | 调用大模型批量拉名词清单，生成/更新资产库 |
| 轨B | 微观单章精制 | 轨A的资产被人工锁定后自动激活 | 1个TXT文件 | 逐章出分镜、生图、配音、合成视频 |

### 3.3 双轨滚动生命周期

```
═══════════════════════════════════════════════════════════
  Loop 1.0（宏观）：扫描并锁定 01-10 章全局资产
═══════════════════════════════════════════════════════════
  导入首批10个TXT → DeepSeek拉出12大科目名词清单
  → WebUI人工审核增删 → 确认锁定资产库
═══════════════════════════════════════════════════════════
  Loop 1.1（微观）：逐章精制 01-10 章内容
═══════════════════════════════════════════════════════════
  第1章 → 分集 → 分镜 → 生图 → 配音 → 合成视频
  → 完成后自动切第2章 → ... → 第10章视频全部导出MP4
═══════════════════════════════════════════════════════════
  Loop 2.0（宏观）：增量更新 11-20 章资产
═══════════════════════════════════════════════════════════
  系统自动捕获第11-20个TXT → DeepSeek读取新文本+已有资产库
  → 把新人物/新场景"追加"填入资产库
═══════════════════════════════════════════════════════════
  Loop 2.1（微观）：逐章精制 11-20 章内容
═══════════════════════════════════════════════════════════
  带着最新增量资产库记忆 → 逐章精制第11章、第12章...
═══════════════════════════════════════════════════════════
  Loop 3.0（宏观）：增量更新 21-30 章资产
═══════════════════════════════════════════════════════════
  ...无限套娃，直到小说全部拍完...
```

### 3.4 双轨的核心价值

| 问题 | 没有双轨时 | 有双轨后 |
|------|-----------|----------|
| 角色换脸 | 第50章的王林和第1章长得完全不同 | 每10章更新一次资产库，定妆照始终锁定 |
| 衣服穿帮 | 前面穿黑袍后面变白袍 | 服装参数在资产库里被死死锁住 |
| 设定吃书 | 前面说练气期后面忘了 | 世界设定在资产库里被持续追踪 |
| 大模型遗忘 | 处理第200章时不知道第50章的伏笔 | 百章航海图提供了长线记忆 |

## 四、整体生产链路总览（19 步）

```
① 小说原文（几百章 TXT 文件）
    ↓
② 题材自动识别 + 视觉风格加载
    ↓ 人工审核
③ 十二大资产科目盘点（名词清单）
    ↓ 人工审核锁定
④ 百章宏观世界线航海图
    ↓
⑤ 十章视听控制总纲
    ↓
⑥ 十二大资产细胞级参数卡片
    ↓ 暂存百章沙盒
⑦ 百章沙盒审计（两步漏斗）
    ↓ 资产晋升 + JSON路标集编译
⑧ 单章物理切肉 + 5大素材合流
    ↓
⑨ 大模型全中文标签分集输出
    ↓
⑩ Python 物理切肉 + JSON派工单编译
    ↓
⑪ 叙事蓝图生成
    ↓ 人工审核
⑫ 剧本生成
    ↓ 人工审核
⑬ 声肉脱水 + 情绪调教派工单
    ↓
⑭ 分镜导演设计（15字段）
    ↓ 人工审核
⑮ 提示词精炼（文生图 + 视频防崩）
    ↓ 人工审核
⑯ 英文提示词编译 + 定妆照路径
    ↓ 人工审核
⑰ 素材生成（API网关 + 自动降级 + 定妆照传参）
    ↓ 图片/视频/配音/BGM/音效
⑱ 自动剪辑合成
    ↓
⑲ 成片输出（MP4 + Markdown + 日志）
```

## 六、融合来源标注

本方案的每个环节都标注了来源：
- **【取自系统1】**：来自第一套系统（通用短剧生产线）
- **【取自系统2】**：来自第二套系统（长篇连载生产线）
- **【两套融合】**：取两套精华合并
- **【新增】**：在原两套基础上新增的环节

**核心融合原则：**
- 系统2管"想清楚拍什么"（编剧+导演+选角）
- 系统1管"怎么拍出来"（摄影+后期+发行）
- 长线防崩靠系统2，生产执行靠系统1
